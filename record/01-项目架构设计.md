## 🎯 需求分析

### 项目目标

为 Hexo 博客添加 AI Agent 对话功能，支持：
- 普通对话
- 知识库问答（RAG）
- 外部搜索
- 多轮对话记忆

### 核心功能

```
用户输入
    ↓
意图识别
    ↓
┌─────────┬─────────┬─────────┐
│ 对话    │ 知识库  │ 搜索    │
│ Agent   │ Agent   │ Agent   │
└─────────┴─────────┴─────────┘
    ↓
生成回答
```

## 🔧 技术选型

### 后端框架

| 选项 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| FastAPI | 异步、自动文档、性能好 | 学习曲线 | ✅ |
| Flask | 简单、灵活 | 异步支持弱 | ❌ |
| Django | 功能全、生态好 | 太重 | ❌ |

### 数据库

| 选项 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| PostgreSQL + pgvector | 一体化、功能强 | 内存占用 | ✅ |
| Milvus | 专为向量设计 | 需额外部署 | ❌ |
| Pinecone | 托管服务 | 付费 | ❌ |

### 缓存

| 选项 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| Redis | 快速、功能丰富 | 需部署 | ✅ |
| 本地缓存 | 简单 | 不持久化 | ❌ |

### LLM

| 选项 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| DeepSeek | 便宜、中文好 | 生态小 | ✅ |
| OpenAI | 功能强、生态好 | 贵 | ❌ |
| 本地模型 | 免费、可控 | 需 GPU | ❌ |

## 📊 数据流设计

### 对话流程

```
用户输入 → API 接收
    ↓
验证 Token
    ↓
限流检查
    ↓
保存用户消息
    ↓
Orchestrator 意图识别
    ↓
路由到对应 Agent
    ↓
Agent 处理（流式输出）
    ↓
保存助手回复
    ↓
返回响应
```

### 知识库检索流程

```
用户查询
    ↓
查询向量化（DashScope）
    ↓
向量相似度搜索（pgvector）
    ↓
相似度过滤（> 0.3）
    ↓
返回 Top K 结果
    ↓
LLM 生成回答
```

## 🎨 设计原则

### 1. 模块化

- 每个 Agent 独立实现
- 通过 Orchestrator 统一调度
- 方便扩展新 Agent

### 2. 异步优先

- 使用 async/await
- 支持流式输出
- 提高并发性能

### 3. 配置驱动

- 环境变量配置
- 支持多环境
- 敏感信息不入库

### 4. 容错设计

- 异常捕获
- 优雅降级
- 用户友好提示

## 📝 总结

### 关键决策

1. **选择 FastAPI**：异步支持好，自动文档，性能优秀
2. **选择 pgvector**：与 PostgreSQL 一体化，运维简单
3. **选择 DeepSeek**：便宜，中文效果好
4. **模块化设计**：方便扩展和维护

### 设计亮点

1. **Orchestrator 模式**：统一调度，解耦 Agent
2. **流式输出**：提升用户体验
3. **会话记忆**：支持多轮对话
4. **容错设计**：优雅处理异常

### 后续优化

1. **RAG 优化**：提升检索准确率
2. **ReAct 模式**：支持工具调用
3. **性能优化**：缓存、并发
4. **监控告警**：日志、指标
